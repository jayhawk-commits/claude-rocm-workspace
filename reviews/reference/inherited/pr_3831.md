## Summary

Introduces a centralized flag system (`FLAGS.cmake` + `therock_flag_utils.cmake`) that automates CMake variable and C preprocessor define propagation to subprojects. Flags are declared in one place and their effects are injected via the existing `project_init.cmake` mechanism. The first flag migrates `THEROCK_KPACK_SPLIT_ARTIFACTS` ‚Üí `THEROCK_FLAG_KPACK_SPLIT_ARTIFACTS`. Also hardcodes the kpack tool path (removing the `THEROCK_KPACK_DIR` cache variable), adds flag state recording in the manifest, supports branch-specific default overrides via `BRANCH_FLAGS.cmake`, and includes documentation.

**Net changes:** +498 lines, -22 lines across 11 files

## Detailed Review

### 1. `cmake/therock_artifacts.cmake` ‚Äî Kpack tool path

Good change. Replaces the user-configurable `THEROCK_KPACK_DIR` with the well-known `THEROCK_ROCM_SYSTEMS_SOURCE_DIR/shared/kpack/` path, which is where kpack now lives. This removes an unnecessary cache variable and aligns with how other tools under `rocm-systems` are referenced.

---

### 2. `cmake/therock_flag_utils.cmake` ‚Äî Core implementation

### üí° SUGGESTION: `ISSUE` parameter is declared but never used

`therock_declare_flag()` accepts an `ISSUE` keyword and stores it in a global property, but nothing reads it ‚Äî not the JSON output, not the status report, not the docs template. Either wire it into `flag_settings.json` / `therock_report_flags()` or remove it to avoid dead code.

### üí° SUGGESTION: JSON generation is fragile for edge cases

The JSON generation in `therock_finalize_flags()` uses string concatenation with `list(JOIN ...)`. This works for the current simple boolean-only schema, but if a flag description or value ever contains quotes or special characters, it could produce invalid JSON. Fine for now since values are only `true`/`false`, but worth noting if the schema expands.

### üí° SUGGESTION: Consider validating `DEFAULT_VALUE` is ON or OFF

`therock_declare_flag()` validates that `DEFAULT_VALUE` is defined but doesn't check it's actually `ON` or `OFF`. Passing an arbitrary string would silently create a truthy/falsy cache var without error.

---

### 3. `cmake/therock_subproject.cmake` ‚Äî Injection point

Clean integration. The `_therock_get_flag_init_contents()` call at the right point in `therock_cmake_subproject_activate()` means flag-injected content appears after default vars and includes but before the file is finalized. No issues.

---

### 4. `FLAGS.cmake` ‚Äî Flag declarations

Clean and well-structured. The KPACK_SPLIT_ARTIFACTS declaration correctly specifies `CMAKE_VARS ROCM_KPACK_ENABLED=ON` scoped to `hip-clr`, which replaces the manual `-DROCM_KPACK_ENABLED=ON` forwarding that was in `core/CMakeLists.txt`.

Note: The docs example includes `ISSUE "https://github.com/ROCm/TheRock/issues/3448"` but the actual declaration in `FLAGS.cmake` omits it. Consider adding it there too if the ISSUE parameter is kept.

---

### 5. `base/CMakeLists.txt` ‚Äî Flag variable propagation to aux-overlay

### ‚öÝÔ∏è IMPORTANT: `THEROCK_FLAG_SETTINGS_FILE` may be empty when flags are OFF

The flag settings file is always generated (even when all flags are OFF), so this should be fine in practice. But `THEROCK_FLAG_SETTINGS_FILE` is set in `therock_finalize_flags()` via `PARENT_SCOPE` ‚Äî if `therock_finalize_flags()` is called from `FLAGS.cmake` which is `include()`d from the top-level `CMakeLists.txt`, the `PARENT_SCOPE` chain needs to reach the scope where `base/CMakeLists.txt` reads it.

Actually, looking more carefully: `FLAGS.cmake` is included at the top-level scope, and `therock_finalize_flags()` is a function that sets `THEROCK_FLAG_SETTINGS_FILE` in `PARENT_SCOPE` ‚Äî which would be the top-level scope. So `base/CMakeLists.txt` (processed via `add_subdirectory`) would see it. This should work correctly.

**Recommendation:** Disregard ‚Äî this is fine on closer inspection.

---

### 6. `build_tools/generate_therock_manifest.py` ‚Äî Manifest integration

Clean implementation. The `--flag-settings` argument is optional, file existence is validated with a clear error, and the JSON is merged under a `"flags"` key. Uses `encoding="utf-8"` for the file read. No issues.

---

### 7. `core/CMakeLists.txt` ‚Äî Migration of manual kpack forwarding

Good cleanup. The manual `-DROCM_KPACK_ENABLED=ON` cmake arg forwarding to hip-clr is removed because the flag system now handles it. The runtime dependency wiring (`_hip_clr_kpack_runtime_deps`) correctly remains as explicit code ‚Äî flags don't auto-include subprojects.

### 8. `docs/development/flags.md` ‚Äî Documentation

Well-written and thorough. Covers architecture, API, branch overrides, manifest integration, and alternatives considered.

Step 4 says "inspect the generated `project_init.cmake` files" ‚Äî could also mention checking the `flag_settings.json` content and verifying the flag appears in the configure status output.

Correct placement. The `/BRANCH_FLAGS.cmake` entry is root-anchored so it won't affect any nested files.
