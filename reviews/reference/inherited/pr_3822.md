## Summary

Adds a new Python script (`build_tools/determine_rocm_test_dependencies.py`) that parses `therock_cmake_subproject_declare` blocks in CMakeLists.txt files to build a package-level dependency graph. Given a list of changed packages, it outputs the changed packages plus their direct downstream dependents (reverse deps) as a JSON array. Includes a unit test file with 3 basic sanity tests.

**Net changes:** +361 lines across 2 new files

---

## Overall Assessment

**⚠️ CHANGES REQUESTED** — The concept is sound, but the CMake parser has a significant coverage gap (misses all `third-party/` subprojects) and silently swallows parsing errors, which will produce incomplete dependency graphs without any indication of failure.

**Strengths:**

- Clear problem statement and motivation
- Clean class structure with `PackageInfo` and `PackageDependencyAnalyzer`
- Good separation between the analyzer class and CLI entry point
- Reverse dependency graph approach is appropriate for this use case
- Provides both CLI and programmatic (`create_analyzer()`) interfaces

**Blocking/Important Issues:**

- Missing `third-party/` in search directories — ~20 subprojects invisible to the analyzer
- Silent `except Exception: pass` violates fail-fast principle
- CMake closing-paren heuristic will misparse nested blocks
- Variable deps (`${...}`) silently skipped with no warning or documentation

## Detailed Review

### 1. `build_tools/determine_rocm_test_dependencies.py`

#### ❌ BLOCKING: Missing `third-party/` search directory

Lines 60-72 hardcode a list of search directories but omit `third-party/`, which contains ~20+ subproject declarations (e.g., `therock-spdlog`, `therock-fmt`, `therock-boost`, `therock-openmpi`, `therock-host-blas`, all sysdeps). These packages appear as RUNTIME_DEPS of other subprojects (e.g., `rocm_smi_lib` depends on `${THEROCK_BUNDLED_LIBDRM}` which resolves to a third-party package).

Without `third-party/`, the analyzer:
- Won't know these packages exist (can't query them with `--changed`)
- Won't build correct reverse-dep edges (if package A depends on third-party B, changing B won't trigger A's tests)

**Required action:** Add `self.therock_dir / "third-party"` to `search_dirs`. Consider also adding `self.therock_dir / "math-libs" / "support"` (contains additional subproject declarations).

Better yet: instead of hardcoding directories, recursively search the entire `therock_dir` for CMakeLists.txt files containing `therock_cmake_subproject_declare`, which would be resilient to new top-level directories being added.

#### ❌ BLOCKING: Silent exception swallowing (fail-fast violation)

Line 80:
```python
except Exception as e:
    # Skip files that can't be read
    pass
```

This violates the [Python style guide fail-fast principle](https://github.com/ROCm/TheRock/blob/main/docs/development/style_guides/python_style_guide.md#fail-fast-behavior). If a CMakeLists.txt file can't be read, the dependency graph will be silently incomplete. No other script in `build_tools/` uses `except Exception: pass`.

**Required action:** Remove the bare except. Let exceptions propagate. If specific exceptions should be tolerated (e.g., encoding errors in non-CMake files), catch them narrowly and log a warning.

#### ⚠️ IMPORTANT: Closing-paren heuristic is fragile

Lines 97-100:
```python
decl_end = content.find("\n)", start_pos)
if decl_end == -1:
    decl_end = len(content)
decl_block = content[start_pos : decl_end + 2]
```

This finds the next `\n)` after the declaration start, but CMakeLists.txt files can have nested function calls or other `\n)` patterns before the actual closing paren of `therock_cmake_subproject_declare`. For example, if a `CMAKE_ARGS` section contains a multi-line `$<...>` generator expression or nested `if()` block, this could truncate the declaration prematurely.

In practice, most `therock_cmake_subproject_declare` blocks don't contain nested parens, so this may work for the current codebase, but it's brittle.

**Recommendation:** Consider a simple paren-depth counter instead: scan forward from `match.start()`, incrementing on `(` and decrementing on `)`, stopping when depth reaches 0.

#### ⚠️ IMPORTANT: Variable dependencies silently skipped

Line 116:
```python
if dep and not dep.startswith("${"):
    pkg_info.runtime_deps.add(dep.lower())
```

Many RUNTIME_DEPS use CMake variables (e.g., `${hipBLASLt_runtime_deps}`, `${THEROCK_BUNDLED_LIBDRM}`, `${optional_profiler_deps}`). These are silently dropped, making the dependency graph incomplete for packages like hipBLASLt, rocprofiler-sdk, rocm_smi_lib, and MIOpen.

Since resolving CMake variables from Python is non-trivial, this is an acceptable limitation, but it should be:
1. Documented in the module docstring or a comment
2. Printed as a warning when `--changed` includes a package with unresolved variable deps

**Recommendation:** Add a comment or log message noting the limitation. Consider maintaining a static mapping for the most important variable deps if full resolution isn't feasible.

#### ⚠️ IMPORTANT: Hardcoded directory list is maintenance-fragile

If a new top-level directory is added to TheRock (which has happened — `dctools` is relatively recent), this script silently ignores it.

**Recommendation:** Scan all subdirectories of `therock_dir` (excluding known non-source dirs like `.git`, `build_tools`, `docs`, `dockerfiles`, `external-builds`) or, simpler, just `rglob("CMakeLists.txt")` from the root.

#### 💡 SUGGESTION: `sys.path.insert` is unnecessary in the main module

Line 29:
```python
sys.path.insert(0, str(Path(__file__).parent))
```

This is at module level but doesn't appear to be used for any imports in this file. The test file has its own `sys.path.insert`. This line can be removed.

#### 💡 SUGGESTION: Type annotation uses deprecated `typing` imports

Line 27 imports `Dict, List, Set, Optional` from `typing`. Since Python 3.9+, the built-in types (`dict`, `list`, `set`) support subscripting directly, and `Optional[X]` can be written as `X | None`. Not blocking since existing TheRock code uses both styles.

### 2. `build_tools/tests/determine_rocm_test_dependencies_test.py`

#### ⚠️ IMPORTANT: Test coverage is minimal

Only 3 tests, all happy-path sanity checks:
- `test_basic_parsing` — one package, one dep
- `test_get_packages_to_test` — two packages, one reverse dep
- `test_create_analyzer` — same as test_basic_parsing via helper

Missing test coverage for:
- **Multiple changed packages** — `get_packages_to_test(["rocblas", "hip-clr"])`
- **Package with no dependents** — verify it returns just itself
- **Unknown package validation** — verify error handling for invalid package names
- **Variable deps** — verify they're handled (skipped or resolved) without error
- **CMake files with no `therock_cmake_subproject_declare`** — should be silently skipped
- **`--list-packages` output format**
- **Edge cases in RUNTIME_DEPS parsing** — comments, blank lines, multi-keyword blocks

**Recommendation:** Add at least tests for multiple changed packages, no-dependent packages, and the variable-dep filtering behavior.

#### 💡 SUGGESTION: Tests could use `tmp_path` pytest fixture

The tests manually create `tempfile.mkdtemp()` and `shutil.rmtree()` in setUp/tearDown. If using pytest (which TheRock uses elsewhere), the `tmp_path` fixture handles this automatically. Not blocking since the tests use unittest style which is also used in the project.

## Recommendations

### ❌ REQUIRED (Blocking):

1. Add `third-party/` to `search_dirs` (or switch to recursive search from root)
2. Remove the bare `except Exception: pass` — let errors propagate or catch narrowly

### ✅ Recommended:

3. Use a paren-depth counter instead of `\n)` heuristic for finding declaration block boundaries
4. Document the `${...}` variable limitation and consider a warning when encountered
5. Replace hardcoded directory list with recursive search to avoid future maintenance
6. Add more test cases (multiple changed packages, no dependents, variable deps)

### 💡 Consider:

7. Remove unused `sys.path.insert` in main module
8. Use modern type annotation syntax (`dict` instead of `Dict`, etc.)

### 📋 Future Follow-up:

9. Integration with CI workflows (mentioned as "next steps" in PR)
10. Static mapping or partial resolution for common CMake variable deps
11. Integration with or deduplication against existing dependency information in `BUILD_TOPOLOGY.toml`

## Conclusion

**Approval Status: ⚠️ CHANGES REQUESTED**

The dependency analysis approach is sound, but the two blocking issues (missing `third-party/` directory and silent error swallowing) will produce silently incomplete dependency graphs. The `third-party/` gap is particularly impactful since many core packages depend on third-party subprojects. After fixing the blocking issues and addressing the important items, this should be a useful tool for optimizing CI test selection.
