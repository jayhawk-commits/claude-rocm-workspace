## Detailed Review

### 1. `build_portable_linux_python_packages.yml` — Inline Bash Complexity

#### ❌ BLOCKING: Conditional logic in `run:` block

The "Fetch artifacts" step (lines 84-108 in the diff) now has an `if/else` conditional:

```yaml
- name: Fetch artifacts
  run: |
    if [ -n "${{ inputs.amdgpu_families }}" ]; then
      # Multi-arch mode: use artifact_manager.py ...
    else
      # Single-family mode: use fetch_artifacts.py ...
    fi
```

Per the [GitHub Actions style guide](https://github.com/ROCm/TheRock/blob/main/docs/development/style_guides/github_actions_style_guide.md#prefer-python-scripts-over-inline-bash), conditionals belong in Python scripts, not inline bash. This logic can't be unit tested.

**Required action:** Move this to a Python script (or extend `artifact_manager.py` / `fetch_artifacts.py` to handle both modes via a flag). The workflow step should be a single `python` command.

#### 💡 SUGGESTION: Shell comment inside multi-line Python arg

```yaml
python ./build_tools/artifact_manager.py fetch \
  --run-id="${{ env.ARTIFACT_RUN_ID }}" \
  --stage=all \
  --amdgpu-families="${{ inputs.amdgpu_families }}" \
  --output-dir="${{ github.workspace }}"
  # NOTE: artifact_manager.py appends /artifacts to --output-dir
```

The `# NOTE:` comment after the last argument (without a backslash continuation) technically works because the last arg has no trailing `\`, but it's fragile and could confuse readers. If the fetch logic moves to a script, this becomes moot. Otherwise, move the comment above the command.

### 2. `upload_python_packages.py` — `find_package_files()` doesn't recurse

#### ⚠️ IMPORTANT: Package validation misses subdirectory files in multi-arch mode

`find_package_files()` uses `dist_dir.glob(pattern)` which only matches at the top level:

```python
def find_package_files(dist_dir: Path) -> list[Path]:
    files = []
    for pattern in ["*.whl", "*.tar.gz", "index.html"]:
        files.extend(dist_dir.glob(pattern))
    return sorted(files)
```

In multi-arch mode, family-specific wheels live in subdirectories (`dist/gfx94X-dcgpu/*.whl`). The function is used in `upload_packages()` for:
1. **Validation** — `if not package_files: raise FileNotFoundError` — this probably won't fail since generic packages still exist at top level, but it's misleading
2. **Logging** — "Found N package files" undercounts

The actual upload via `backend.upload_directory()` is recursive and handles subdirs correctly, so uploads aren't affected.

**Recommendation:** Use `dist_dir.rglob(pattern)` or add a `recursive` parameter to `find_package_files()` for multi-arch mode, so the validation and logging reflect what's actually being uploaded.

### 3. `generate_package_index.py` — New module

#### 💡 SUGGESTION: Warning via `print()` instead of logging

```python
if not family_dirs:
    print(f"Warning: No family subdirectories found in {dist_dir}")
    print("This may not be a multi-arch build.")
    return
```

The "Warning" case silently returns, which is arguably the right behavior (it's called from `generate_index()` which is called unconditionally). But the `print()` calls are inconsistent with the `log()` pattern used in the calling script. Consider using `logging.warning()` or raising an exception if this state is truly unexpected.

### 4. `artifact_manager.py` — `--amdgpu-families=all` mode

#### 💡 SUGGESTION: Prefix matching could be over-inclusive

```python
matched_filenames = [
    filename
    for filename in sorted(available)
    if any(filename.startswith(f"{name}_") for name in inbound)
]
```

This matches any available artifact whose name starts with an inbound artifact name followed by `_`. This relies on the naming convention `{name}_{component}_{family}.tar.zst`. If two artifacts share a prefix (e.g., `rocm-core` and `rocm-core-extended`), this could match unintended artifacts. In practice, TheRock's naming conventions make this unlikely, and the existing `find_available_artifacts` function uses similar prefix logic.

**`ci_windows.yml`**: Calls `test_rocm_wheels.yml` — the new `if: test_runs_on != ''` guard is safe since `ci_windows.yml` always passes a non-empty `test_runs_on`.
