## Summary

This PR migrates PyTorch testing from direct pytest invocation (via `run_pytorch_tests.py`) to using PyTorch's own `test/run_test.py` harness (via a new `run_pytorch_tests_full.py`), with 4-shard parallel execution and test-time-based sharding. A new standalone workflow `test_pytorch_wheels_full.yml` is also added for independent dispatch.

**Net changes:** +679 lines, -10 lines across 3 files

---

## Overall Assessment

**⚠️ CHANGES REQUESTED** — The new `run_pytorch_tests_full.py` script is well-structured and the approach of using PyTorch's native `run_test.py` harness is sound. However, there are significant concerns about the in-place modification of the existing `test_pytorch_wheels.yml` workflow (which existing callers depend on) and some correctness issues in the new script.

**Strengths:**

- Good use of PyTorch's own `run_test.py` harness for proper sharding, test selection, and config handling
- Clean CLI design with env-var fallbacks for all sharding parameters
- Proper `AMDGPU_FAMILY_TO_BUILD_ENV` mapping for test timing data
- `--dry-run` flag for debugging test selection
- Good comments explaining why `PYTORCH_TEST_RUN_EVERYTHING_IN_SERIAL` is needed on 1-GPU runners

**Issues:**

- Modifying `test_pytorch_wheels.yml` in-place changes behavior for existing callers
- Duplicate workflow file is largely copy-paste
- Import style issue in new script

## Detailed Review

### 1. test_pytorch_wheels.yml — Breaking Change to Existing Callers

### ❌ BLOCKING: In-place modification changes existing caller behavior

The existing `test_pytorch_wheels.yml` is called by:
- `build_portable_linux_pytorch_wheels.yml` (line 338)
- `build_windows_pytorch_wheels.yml` (line 375)

Neither caller passes `tests_to_include`, so the default applies — but the bigger issue is that this PR changes the Linux test step from `run_pytorch_tests.py` (pytest-based, 6 hardcoded test files) to `run_pytorch_tests_full.py` (run_test.py-based, broader test suite). It also adds 4-shard matrix execution where previously there was a single job.

This means every existing pytorch wheel build will now:
1. Run 4x as many CI jobs (4 shards instead of 1)
2. Use a completely different test harness
3. Potentially run a much broader set of tests (depending on `TESTS_TO_INCLUDE` behavior in `run_test.py`)

**Required action:** Either:
- (a) Only modify the new `test_pytorch_wheels_full.yml` and leave `test_pytorch_wheels.yml` unchanged, OR
- (b) Update the callers (`build_portable_linux_pytorch_wheels.yml`, `build_windows_pytorch_wheels.yml`) to explicitly pass the parameters they need, and document the behavioral change in the PR description

Option (a) is strongly preferred — keep the existing workflow as-is and iterate on the new `_full` workflow independently until it's proven stable. Then migrate callers in a follow-up PR.

### ⚠️ IMPORTANT: pytorch_git_ref default changed from "release/2.7" to "release/2.9"

The default `pytorch_git_ref` was bumped from `release/2.7` to `release/2.9` in both `workflow_dispatch` and `workflow_call` inputs. The callers pass this explicitly, so it only affects manual `workflow_dispatch` runs, but this is a semantic change that should be called out in the PR description.

### ⚠️ IMPORTANT: `pip freeze` removed from Install test requirements

The original workflow had `pip freeze` after installing test requirements. This was useful for debugging dependency issues in CI logs. The new workflow removes it.

**Recommendation:** Keep `pip freeze` (or `python -m pip list`) after installing requirements for debuggability.

### ⚠️ IMPORTANT: Additional requirements file added without justification

The step now also installs `pytorch/.ci/docker/requirements-ci.txt`. This adds PyTorch CI's internal dependencies (which may be substantial). The PR doesn't explain which specific packages from this file are needed or why.

**Recommendation:** Document why this is needed (presumably for `run_test.py` imports). Consider whether a targeted `pip install` of only the needed packages would be more appropriate to avoid unexpected dependency changes.

### 2. test_pytorch_wheels_full.yml — Near-Complete Duplication

### ⚠️ IMPORTANT: Workflow is 95% copy-paste of test_pytorch_wheels.yml

The new `test_pytorch_wheels_full.yml` (236 lines) is almost identical to the modified `test_pytorch_wheels.yml`. The only real differences are:
- Workflow name ("Full Suite" vs regular)
- `run-name` text
- No `workflow_call` `repository`/`ref` inputs... wait, it does have them.

Actually comparing them, they are functionally identical after the PR's modifications to `test_pytorch_wheels.yml`. This creates a maintenance burden — any future fix to one must be applied to both.

**Recommendation:** If the intent is to have a separate "full" workflow, it should call `test_pytorch_wheels.yml` via `workflow_call` rather than duplicating it. Alternatively, if `test_pytorch_wheels.yml` is being migrated to the new behavior, the `_full` variant is unnecessary.

### 3. run_pytorch_tests_full.py — New Test Runner Script

### ❌ BLOCKING: Star import in skip_tests but non-star in new script

The existing `run_pytorch_tests.py` uses:
```python
from skip_tests.create_skip_tests import *
```

The new `run_pytorch_tests_full.py` uses:
```python
from skip_tests.create_skip_tests import get_tests
```

The explicit import is actually *better* style, but verify that `get_tests` is the only symbol needed. If `create_skip_tests` sets module-level state that `run_pytorch_tests.py` was relying on via `import *`, this would be a subtle bug.

**Required action:** Verify that the explicit import is sufficient — check if `create_skip_tests` exports anything besides `get_tests` that has side effects or is needed by the script.

### 💡 SUGGESTION: `subprocess.run` without `check=True`

Line 336 (approximately):
```python
result = subprocess.run(cmd, cwd=str(args.pytorch_dir))
```

This intentionally doesn't use `check=True` because test failures are expected and the return code is propagated. This is correct for this use case, but consider adding a brief comment explaining this is intentional.

### 💡 SUGGESTION: `force_exit_with_code` uses `Path.write_text` vs original's `open()`

The new script uses `retcode_file.write_text(str(retcode_int))` which is cleaner than the original's `with open(...)`. This is fine but note the exit code file name differs: `run_pytorch_tests_full_exit_code.txt` vs `run_pytorch_tests_exit_code.txt`. The Windows workflow steps in `test_pytorch_wheels_full.yml` still reference the old file name `run_pytorch_tests_exit_code.txt`.

Wait — looking again at the `_full` workflow, the Windows step references `run_pytorch_tests_exit_code.txt` but the script writes to `run_pytorch_tests_full_exit_code.txt`. This is actually fine because the Windows steps in the `_full` workflow call `run_pytorch_tests.py` (the old script), not `run_pytorch_tests_full.py`. But this is confusing — why does the "full" workflow call the old script for Windows?

### ⚠️ IMPORTANT: Windows steps in test_pytorch_wheels_full.yml call the old run_pytorch_tests.py

The "full" workflow's Windows test step runs `run_pytorch_tests.py` (the old pytest-based runner), not `run_pytorch_tests_full.py`. This is inconsistent with the Linux step. If this is intentional (because `run_test.py` doesn't work on Windows), it should be documented with a comment.

### 💡 SUGGESTION: `--keep-going` flag

The script always passes `--keep-going` to `run_test.py`. This is probably correct for CI (don't stop on first test file failure), but it's hardcoded with no CLI override. Consider making it configurable or at least documenting why it's always on.

### 4. Environment Setup

### 💡 SUGGESTION: `safe.directory` and `fetch.parallel` added without explanation

The "Adjust git config" step adds `git config --global --add safe.directory $PWD` and `git config fetch.parallel 10`. The comment says "safe.directory must be set before Runner Health Status" but doesn't explain why the health status check needs it, or why `fetch.parallel 10` is beneficial.

### 💡 SUGGESTION: `export_test_times.py` step uses pushd/popd

The "Export test times for sharding" step:
```yaml
run: |
  pushd external-builds/pytorch/pytorch
  PYTHONPATH=. python tools/stats/export_test_times.py
  popd
```

The comment explains the PYTHONPATH requirement well. The `pushd`/`popd` pattern is fine for this simple case.

The core approach (using PyTorch's `run_test.py` with sharding) is solid and the new `run_pytorch_tests_full.py` script is well-structured.
