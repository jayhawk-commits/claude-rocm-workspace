## Summary

Adds four new GitHub Actions workflows to rocm-systems for RCCL CI testing after
migrating RCCL from `rocm/rccl` to `rocm-systems/projects/rccl`. The workflows
include a top-level CI orchestrator, a Linux build workflow, and single-node /
multi-node test workflows. The orchestrator inspects `therock_configure_ci.py`
output to skip Linux CI when RCCL isn't affected.

**Net changes:** +407 lines across 4 new files

---

## Overall Assessment

**⚠️ CHANGES REQUESTED** — The build workflow is missing the
`THEROCK_ROCM_SYSTEMS_SOURCE_DIR` CMake option, which means it builds from the
pinned TheRock commit's submodules rather than the rocm-systems checkout. This
completely defeats the purpose of running CI in rocm-systems. Additional issues
include inconsistent commit pins, command injection pattern, and a `BUILD_DIR`
mismatch.

**Strengths:**

- Smart conditional triggering: only runs RCCL CI when `projects/rccl` is affected
- Good use of OIDC for AWS credentials (role-to-assume)
- Appropriate concurrency groups
- Minimal permissions at workflow level

**Blocking Issues:**

- Missing `THEROCK_ROCM_SYSTEMS_SOURCE_DIR` — builds wrong source
- Inconsistent TheRock commit pinning (two different commits across 3 workflows)
- Potential command injection in `rccl_check` step
- `BUILD_DIR` env var doesn't match hardcoded paths in subsequent steps
- Inconsistent `actions/checkout` versions

## Detailed Review

### 1. therock-rccl-ci.yml (Orchestrator) / therock-rccl-ci-linux.yml (Build)

#### ❌ BLOCKING: Missing `THEROCK_ROCM_SYSTEMS_SOURCE_DIR` — builds wrong source

The entire purpose of running CI in rocm-systems is to test the *rocm-systems*
source (e.g., `projects/rccl`). The existing `therock-ci-linux.yml` passes:

```yaml
extra_cmake_options: "-DTHEROCK_ROCM_SYSTEMS_SOURCE_DIR=../ ${{ inputs.cmake_options }}"
```

This tells the TheRock build to overlay the rocm-systems checkout on top of
TheRock's submodule sources. Without this flag, the build uses whatever source
is in the pinned TheRock commit's submodules — completely ignoring the
rocm-systems checkout that triggered the CI run.

The new `therock-rccl-ci.yml` passes only:

```yaml
extra_cmake_options: >
  -DTHEROCK_ENABLE_ALL=OFF
  -DTHEROCK_BUILD_TESTING=ON
  -DTHEROCK_BUNDLE_SYSDEPS=ON
  -DTHEROCK_ENABLE_COMM_LIBS=ON
  -DTHEROCK_ENABLE_ROCPROFV3=ON
  -DTHEROCK_ENABLE_MPI=ON
```

No `THEROCK_ROCM_SYSTEMS_SOURCE_DIR`. This means every CI run builds and tests
an unchanging snapshot of RCCL from the pinned TheRock commit, regardless of
what changes were made in the PR.

**Required action:** Add `-DTHEROCK_ROCM_SYSTEMS_SOURCE_DIR=../` to
`extra_cmake_options` in the `therock-ci-linux` job in `therock-rccl-ci.yml`.

### 2. therock-rccl-ci.yml (Orchestrator)

#### ⚠️ IMPORTANT: Command injection pattern in `rccl_check` step

```yaml
echo '${{ steps.configure.outputs.projects }}'

if echo '${{ steps.configure.outputs.projects }}' | \
   jq -e '.[] | select(.project_to_test | test("rccl"))' > /dev/null; then
```

The `${{ }}` expression is interpolated by GitHub Actions before the shell runs,
so single quotes don't protect against injection. If `projects` output contains
shell metacharacters, this could break or execute unintended commands.

The risk is low here since the data comes from a script the workflow itself runs
(not external user input), but it's still best practice to use env vars.

**Recommendation:** Pass through an environment variable:
```yaml
- name: "Check if RCCL project is affected"
  id: rccl_check
  env:
    PROJECTS_JSON: ${{ steps.configure.outputs.projects }}
  run: |
    echo "Projects JSON:"
    echo "$PROJECTS_JSON"

    if echo "$PROJECTS_JSON" | \
       jq -e '.[] | select(.project_to_test | test("rccl"))' > /dev/null; then
      echo "run_linux_ci=true" >> "$GITHUB_OUTPUT"
    else
      echo "run_linux_ci=false" >> "$GITHUB_OUTPUT"
    fi
```

### 3. therock-rccl-ci-linux.yml (Build)

#### ⚠️ IMPORTANT: Inconsistent TheRock commit pinning

Three workflows pin TheRock checkout to different commits:

| Workflow | Commit | Comment |
|----------|--------|---------|
| `therock-rccl-ci-linux.yml` | `9a8534f5e38c8c5e20b1a92185aca578c2840a7d` | "02-05-2026" |
| `therock-rccl-test-packages-multi-node.yml` | `9a8534f5e38c8c5e20b1a92185aca578c2840a7d` | "02-05-2026" |
| `therock-rccl-test-packages-single-node.yml` | `5523139d860b6430aa24baf044e6dfa59921df60` | "02/11/2026" |

The single-node workflow uses a different (newer) commit than the other two.
This could lead to mismatched build vs. test behavior.

**Recommendation:** Align all workflows to the same TheRock commit. Consider
using a workflow-level env var or input to make the pinned ref easier to update.

#### ⚠️ IMPORTANT: `BUILD_DIR` env var vs. hardcoded paths

The "Configure Projects" step sets `BUILD_DIR: build`:
```yaml
env:
  BUILD_DIR: build
```

But subsequent steps hardcode `TheRock/build`:
```yaml
- name: Build therock-dist
  run: cmake --build TheRock/build

- name: Build therock-archives
  run: cmake --build TheRock/build --target therock-archives
```

If `build_configure.py` reads `BUILD_DIR` to determine where to configure, the
build directory it creates may not match what the cmake commands expect.

**Recommendation:** Either use `TheRock/build` consistently (including in
`BUILD_DIR`), or use `$BUILD_DIR` in the cmake commands.

#### 💡 SUGGESTION: Commented-out `if` condition in Report step

```yaml
- name: Report
  #if: ${{ !cancelled() }}
  run: |
```

This commented-out condition suggests incomplete work. The Report step should
probably run on failure/cancellation too (for debugging), which is what the
commented condition would enable.

**Recommendation:** Either uncomment `if: ${{ !cancelled() }}` or remove the
comment to avoid confusion.

#### 💡 SUGGESTION: `Post Build Upload` runs on forks without AWS credentials

```yaml
- name: Configure AWS Credentials for non-forked repos
  if: ${{ always() && !github.event.pull_request.head.repo.fork }}
  ...
- name: Post Build Upload
  if: always()
```

On fork PRs, AWS credentials are skipped but the upload step still runs. This
will likely fail silently (or noisily) because credentials aren't configured.

**Recommendation:** Add the fork check to the upload step too, or ensure the
upload script gracefully handles missing credentials.

### 4. Consistency Across Workflows

#### ⚠️ IMPORTANT: Inconsistent `actions/checkout` versions

| Workflow | SHA | Version |
|----------|-----|---------|
| `therock-rccl-ci.yml` | `08c6903cd8c0fde910a37f88322edcfb5dd907a8` | v5.0.0 |
| `therock-rccl-ci-linux.yml` | `08c6903cd8c0fde910a37f88322edcfb5dd907a8` | v5.0.0 |
| `therock-rccl-test-packages-multi-node.yml` | `11bd71901bbe5b1630ceea73d27597364c9af683` | v4.2.2 |
| `therock-rccl-test-packages-single-node.yml` | `11bd71901bbe5b1630ceea73d27597364c9af683` | v4.2.2 |

Two different checkout action versions are in use. While both likely work, this
is a maintenance burden and could lead to subtle behavioral differences.

**Recommendation:** Standardize on one version across all workflows.

### 5. therock-rccl-test-packages-single-node.yml

#### 💡 SUGGESTION: Bare `pytest` invocation

```yaml
run: |
  pytest ./build_tools/github_actions/test_executable_scripts/test_rccl.py -v -s \
    -k "not test_rccl_correctness_tests" \
    --log-cli-level=info
```

Using bare `pytest` rather than `python -m pytest`. This works if pytest is
installed in the active Python environment, but `python -m pytest` is more
explicit about which Python interpreter is used.

## Recommendations

### ❌ REQUIRED (Blocking):

1. **Add `THEROCK_ROCM_SYSTEMS_SOURCE_DIR`** — without this, the CI builds the
   wrong source and doesn't test the PR's changes at all

### ✅ Recommended:

1. **Fix command injection pattern** in `rccl_check` step — use env var instead of `${{ }}` in shell
2. **Align TheRock commit pins** — all 3 workflows should reference the same commit
3. **Resolve `BUILD_DIR` mismatch** — ensure env var matches hardcoded paths
4. **Standardize `actions/checkout` version** across all 4 workflows

### 💡 Consider:

1. Uncomment or remove the `#if:` condition in the Report step
2. Add fork check to `Post Build Upload` step
3. Use `python -m pytest` instead of bare `pytest`

### 📋 Future Follow-up:

1. Consider parameterizing the TheRock commit pin (e.g., as a workflow input with
   a default) to make updates easier across all workflows
2. Consider whether the old workflow files (`therock-ci.yml`, etc.) need to be
   deleted or updated to avoid confusion with the new `therock-rccl-*` variants

## Conclusion

**Approval Status: ⚠️ CHANGES REQUESTED**

The workflow structure is reasonable and the conditional RCCL triggering is a good
approach. However, the missing `THEROCK_ROCM_SYSTEMS_SOURCE_DIR` flag means the
CI doesn't actually test rocm-systems source changes — it builds an unchanging
snapshot from the pinned TheRock commit. This is a fundamental correctness issue
that needs an immediate follow-up fix. The other issues (inconsistent commit pins,
command injection pattern, `BUILD_DIR` mismatch) should be addressed at the same
time.
