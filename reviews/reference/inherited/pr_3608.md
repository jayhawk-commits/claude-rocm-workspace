## Summary

Fixes three related bugs that made multi-arch Python packaging (e.g.
gfx120X-all + gfx94X-dcgpu built together) produce broken or incomplete
packages:

1. **Library wheel dedup bug**: A shared `PopulatedFiles` on `Parameters`
   caused whichever family iterated first to claim all shared relpaths,
   leaving subsequent families with empty library packages. Fix: each
   `PopulatedDistPackage` now has its own `self.files`.

2. **Single devel wheel bug**: The devel package used
   `params.default_target_family` for artifact filtering, so only one
   family's headers/libs ended up in the tarball. Fix: one devel wheel per
   target family, each filtering by `self.target_family`.

3. **Single meta sdist bug**: The meta package baked one arbitrary default
   family. Fix: one meta sdist per target family in multi-arch builds, with
   `restrict_families=True` to scope `_dist_info.py` to that family.

**Net changes:** +716 lines, -49 lines across 4 files

## Detailed Review

### 1. py_packaging.py — Core Fix: Per-Instance `PopulatedFiles`

#### ✅ Correct: `PopulatedFiles` moved to `PopulatedDistPackage`

The central change is moving `self.files = PopulatedFiles()` from `Parameters.__init__`
to `PopulatedDistPackage.__init__`. Each package tracks its own materialized files
independently, eliminating the cross-architecture interference bug.

All references updated consistently:
- `self.params.files` → `self.files` in `populate_runtime_files`, `_populate_runtime_symlink`, `_populate_file`
- `self.params.files.soname_aliases` → `self.files.soname_aliases`
- `self.params.files.mark_populated` → `self.files.mark_populated`

#### ✅ Correct: Package self-registration

`self.params.populated_packages.append(self)` at the end of `populate_runtime_files()`
ensures only fully-populated packages are searchable. Packages that don't call
`populate_runtime_files()` (like `meta`) are correctly excluded.

#### ✅ Correct: `_find_populated()` and `_find_soname_alias()`

Target-family filtering logic is sound:
- `self.target_family is None` (generic devel) → matches any package
- `pkg.target_family is None` (generic core) → always matched
- Both non-None and different → skip (correct cross-family isolation)

#### ⚠️ IMPORTANT: Stale comment in `_devel_artifact_filter`

The filter now uses `self.target_family` instead of `self.params.default_target_family`,
but the comment still says "We only materialize the default target family for devel
packages." This is misleading since each devel package now filters to *its own*
target family.

**Recommendation:** Update the comment to reflect the new behavior, e.g.:
```python
# Only include artifacts for this devel package's target family.
```

### 2. py_packaging.py — `restrict_families` and `dist_info_base`

#### ✅ Good: Base/full dist_info split

Splitting `dist_info_contents` into `dist_info_base_contents` (version + nonce
only) and `dist_info_contents` (base + families) is clean. The restricted
packages start from base and append only their own family, avoiding `.clear()`.

#### ✅ Correct: Guard condition

```python
if restrict_families and target_family is not None:
```

Handles the edge case where `restrict_families=True` is passed with
`target_family=None` — falls through to the full dist_info. The test
`test_restrict_families_ignored_when_target_family_is_none` covers this.

### 3. build_python_packages.py — Build Orchestration

#### ⚠️ IMPORTANT: Ternary operator precedence could be clearer

Two lines use a ternary conditional on a Path expression:

```python
dist_dir=args.dest_dir / "dist" / target_family if multi_arch else None,
```

Python operator precedence makes this parse correctly as
`(args.dest_dir / "dist" / target_family) if multi_arch else None`, but the
intent would be clearer with explicit parentheses. Appears at two locations.

**Recommendation:** Add parentheses around the Path expression for clarity.

#### ✅ Correct: Sorted iteration

`sorted(params.all_target_families)` ensures deterministic iteration order,
fixing the non-determinism root cause.

#### ✅ Good: Build ordering

Building non-devel, non-meta wheels first prevents the default
`dest_dir.iterdir()` scan from accidentally including later-created package dirs.

#### 💡 SUGGESTION: Explicit `package_dirs` for non-devel builds

The correctness of the first `build_packages` call depends on devel/meta package
directories not existing yet. Passing `package_dirs` explicitly would guard
against future reordering regressions.

### 4. py_packaging.py — `build_packages()` Refactoring

#### ✅ Correct: New optional parameters

Adding `package_dirs` and `dist_dir` parameters with sensible defaults maintains
backward compatibility while enabling per-family builds.

#### ✅ Good: Dead code removal

The old double-assignment of `build_args` is cleaned up.

### 5. Tests — py_packaging_test.py (553 lines)

#### ✅ Good: Unit tests (`PopulatedFilesTest`)

Tests per-instance isolation directly:
- `test_two_instances_are_independent` — regression test for the bug
- `test_soname_aliases_are_per_instance`
- `test_mark_populated_raises_on_duplicate`

#### ✅ Good: Integration tests (`MultiArchPackagingTest`)

Uses real artifact directories with `.txt` file stand-ins. Covers:
- Shared relpaths across architectures
- Package self-registration
- Cross-package search with `_find_populated` and `_find_soname_alias`
- Target-family filtering (matching family, different family, generic fallback)

#### ✅ Good: `RestrictFamiliesTest`

Covers `restrict_families` behavior:
- Per-family restriction works for each family
- No restriction lists all families
- Single-arch build (no-op)
- `target_family=None` edge case
- No dead writes (`.clear()`, stray families) in generated code

#### 💡 SUGGESTION: Consider a negative test for `_find_populated` with wrong-only family

`test_find_populated_prefers_matching_target_family` tests the correct family is
returned. Consider also testing that when *only* a wrong-family package has a
relpath, `_find_populated` returns `None`.
