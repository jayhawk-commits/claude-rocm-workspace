## Summary

This PR adds multi-architecture native Linux package building (DEB/RPM) to TheRock's CI. The key changes:

1. **New workflow** `multi_arch_build_native_linux_packages.yml` — a reusable workflow that fetches artifacts for all GPU families, builds unified packages, and uploads to S3.
2. **Wiring into `multi_arch_ci_linux.yml`** — adds `compute_native_package_versions` and `build_native_packages` jobs.
3. **`build_package.py` enhancements** — multi-value `--target`, `--enable-multi-arch` flag, per-architecture iteration loop, explicit failed-package tracking.
4. **`packaging_utils.py` additions** — `GFX_GENERIC` constant, `resolve_versioned_dependencies()`, `get_dependency_list_for_multiarch()`, `expand_metapackage_to_all_archs()`, multi-arch logic in `filter_components_fromartifactory()` and `update_package_name()`.
5. **`upload_package_repo.py`** — new `--s3-prefix` and `--job ci` support; `--amdgpu-family` made optional for backward compat.
6. **`packaging_summary.py`** — replaces hardcoded `2x` multiplier with explicit `failed` list tracking.
7. **`package.json`** — adds `rccl-tests` entry (flagged by existing reviewer).

**Net changes:** +710 lines, -68 lines across 7 files.

## Detailed Review

### 1. `build_package.py` — Failed package double-counting

### :warning: IMPORTANT: Failed package list may double-count entries

In the build loop (around the `run()` function), when a package fails for ALL architecture variants:

```python
for gfxarch in loop_list:
    ...
    else:
        # Add failed architecture variant to failed list
        variant_name = (
            f"{pkg_name}-{gfxarch}" if gfxarch != default_gfx_arch else pkg_name
        )
        failed_pkglist.append(variant_name)

if not pkg_built:
    failed_pkglist.append(pkg_name)
```

If a package fails for the generic arch (where `gfxarch == default_gfx_arch`), `pkg_name` gets appended in the inner `else` block, AND again in the `if not pkg_built` block — resulting in `pkg_name` appearing twice in `failed_pkglist`.

**Recommendation:** Remove the `if not pkg_built: failed_pkglist.append(pkg_name)` block, since individual variant failures are already tracked. Or restructure so the outer check only fires when the inner loop didn't already record it.

---

### 2. `multi_arch_build_native_linux_packages.yml` — Dead `strategy` block

### :bulb: SUGGESTION: Remove meaningless `strategy` block

```yaml
build_native_packages:
    strategy:
      fail-fast: false
```

This job has `strategy: fail-fast: false` but no `matrix`. Without a matrix, `strategy` does nothing. It should be removed to avoid confusion.

---

### 3. `multi_arch_build_native_linux_packages.yml` — Dead `run-name`

### :bulb: SUGGESTION: Remove `run-name` from reusable workflow

```yaml
run-name: Build multi-arch native Linux packages (${{ inputs.dist_amdgpu_families }}, ...)
```

For workflows triggered only via `workflow_call`, `run-name` is ignored (the caller's run name is used instead). This line has no effect.

### 5. `build_package.py` — `dh_strip` override uses string instead of boolean

### :bulb: SUGGESTION: Use consistent type for `disable_dh_strip`


### 8. `multi_arch_ci_linux.yml` — Hardcoded `release_type: ""`

### :bulb: SUGGESTION: Consider parameterizing `release_type`

The new `build_native_packages` job hardcodes `release_type: ""` which means all multi-arch CI package builds go to the CI artifacts bucket. This is presumably intentional for now, but worth noting that release workflows will need a separate caller or parameterization later.

---

### 9. `packaging_utils.py` — Complex dependency resolution logic

### :warning: IMPORTANT: `resolve_versioned_dependencies` has 4 branches with subtle semantics

The function has a complex 4-way conditional:
1. Generic meta + multi-arch: deps already expanded, just suffix
2. Arch-specific meta + multi-arch: preserve arch, then suffix
3. Non-meta gfx-specific + multi-arch: split dep_list[0] vs dep_list[1:]
4. Normal path

Branch 3 assumes `dep_list[0]` is always the versioned dependency and `dep_list[1:]` are gfxarch dependencies. This ordering is enforced by `get_dependency_list_for_multiarch()` which returns `[pkg_name] + gfxarch_deps`, but the coupling between these two functions is implicit and fragile.

**Recommendation:** Add a comment or assertion in `resolve_versioned_dependencies` documenting the expected structure of `dep_list` for branch 3, or consider using a typed structure instead of positional list semantics.

---

### 10. `multi_arch_build_native_linux_packages.yml` — Fetch artifacts step

### :bulb: SUGGESTION: Quote variable in word-splitting expansion

```bash
FAMILIES=(${DIST_AMDGPU_FAMILIES//;/ })
```

This relies on word splitting to create the array. If any family name ever contained spaces (unlikely but possible), this would break. The safer pattern:

```bash
IFS=';' read -ra FAMILIES <<< "${DIST_AMDGPU_FAMILIES}"
```
